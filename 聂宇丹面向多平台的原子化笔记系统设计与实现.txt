摘要 
本论文研究的可视化系统利用基于 WebGL 的 3D 图形 API 库 Three.js，将自动驾驶系统中各模块的输出数据以 3D 图形的表达形式实时地渲染在三维虚拟场景中，实现了自动驾驶系统运行过程的三维可视化，让自动驾驶变得易于观察和理解，使得自动驾驶研发人员在研发测试过程中能够快速准确的了解各自动驾驶模块和系统整体的运行状况。为实现该可视化系统，本研究做了如下工作：
首先，研究了开发自动驾驶可视化系统所需的基础理论和相关技术，包括 Web 开发技术，数据可视化基础理论，自动驾驶系统，WebGL 技术。其中，重点研究了 WebGL技术，对 WebGL 程序的组成以及 WebGL 中所涉及的计算机图形学基础知识进行了研究，为后续可视化系统的开发奠定了坚实的理论基础。 
其次，本论文围绕 3D 自动驾驶场景、人机交互、模块控制和数据图表显示等几个方面进行了详细的功能需求分析和设计，并对功能进行了实现。 
接着，通过对浏览器底层运行和渲染机制的分析，找出影响系统性能的因素，并结合自动驾驶三维场景的特点对系统实现进行了优化。本文创新性地使用了基于模型缓存的三维场景渲染优化算法，提升了三维场景的渲染速度，为用户带来了流畅的三维动画体验，同时也为自动驾驶可视化软件的性能优化提供了一种方法。 
最后，论文对本课题的研究工作进行了总结，指出了本文中的可视化系统当前存在的不足之处，并对后续的研究进行了展望。

ABSTRACT
The visualization system studied in this subject uses the WebGL-based 3D graphics API library Three.js to render the output data of each module in the automatic driving system in the form of 3D graphics in real time in the three-dimensional virtual scene, realizing  the operation process of the automatic driving system. The 3D visualization makes autonomous driving  easy  to  observe  and  understand,  so  that  autonomous  driving  R&D  personnel  can quickly and accurately understand the overall operating status of each autonomous driving module and system during the R&D and testing process. In order to realize the visualization system, this research has done the following work:
First, the basic theory and related technologies required for the development of automatic driving visualization system are studied, including Web development technology, basic theory of data visualization, automatic driving system, and WebGL technology. Among them, the focus is on WebGL technology, the composition of WebGL programs and the basic knowledge of computer graphics involved in WebGL are studied, which lays a solid theoretical foundation for the development of subsequent visualization systems.
Secondly, this thesis analyzes and designs the functional requirements in detail around the 3D automatic driving scene, human-computer interaction, module control and data chart display, and implements the functions. 
Then, through the analysis of the underlying operation and rendering mechanism of the browser, the factors affecting the performance of the system are found out, and the system implementation is optimized according to the characteristics of the 3D scene of autonomous driving. This thesis innovatively uses the model cache-based 3D scene rendering optimization algorithm,  which  improves  the  rendering  speed  of  3D  scenes,  brings  users  a  smooth  3D animation  experience,  and  also  provides  a  method  for  the  performance  optimization  of autonomous driving visualization software.
Finally, the thesis summarizes the research work of this topic, points out the current shortcomings of the visualization system in this thesis, and looks forward to the follow-up research.

Key words:  Autopilot; 3D visualization; Web; WebGL; Three.js

1绪论
1.1课题背景
随着数字化时代的深入发展和移动互联网的广泛普及，信息以前所未有的速度被生产、传播和消费。个人信息管理已成为现代社会不可或缺的关键技能，特别是对于知识工作者、在校学生、软件开发者及科研人员等信息高度依赖的群体而言，有效的个人信息管理是提升工作效率和生活质量的核心挑战。然而，传统的笔记应用和信息管理工具在应对当前多设备、多场景、多任务并行的复杂使用环境时，日益显得力不从心。许多用户不得不在桌面电脑、笔记本电脑、平板电脑和智能手机之间频繁切换，但现有的笔记应用在这一过程中暴露出一系列显著不足。首先，这些工具往往存在信息组织僵化的问题，大多采用“笔记本-笔记”的层级结构或自由画布模式，信息以“文档”或“页面”为基本单位，难以支持更细粒度的信息操作和灵活重组。其次，许多应用虽然提供多端版本，但在不同操作系统上的功能、界面和交互体验差异巨大，割裂了用户的使用流程。更为严重的是，在多设备间实现数据实时、准确的同步是巨大挑战，用户时常遭遇数据延迟、版本冲突甚至信息丢失的问题，尤其是在网络环境不佳时。因此，本研究的目的在于设计一种新型的个人信息管理系统，它应能打破传统笔记的束缚，提供更灵活的信息组织方式，并实现跨平台无缝、可靠的数据同步和一致的使用体验。
1.2目的与意义
1.2.1研究目的
针对上述背景中传统笔记应用的局限性，本研究的核心目的在于设计并实现一款面向多平台的原子化笔记系统。该系统旨在解决现有工具在信息灵活性、跨平台同步和个性化需求方面的不足。为实现此目的，本研究引入了“原子化笔记”的理念。本研究定义的“原子化”并非指代文档内部的块级编辑，而是借鉴其核心思想，将“每一条笔记或待办事项本身”视为一个独立的、不可再分的“信息原子”。
其目的与意义包括：
实现原子化信息管理： 将笔记内容彻底拆分为原子化单元，支持灵活组合和重用。
提供多平台无缝同步： 支持 Web、桌面应用、移动端等多种形态，确保数据在多设备间的实时、一致同步。
构建离线优先架构： 基于 PWA（Progressive Web App）技术，确保在网络环境不佳或离线时仍能正常访问和编辑。
集成轻量化管理功能：为每个“原子”（卡片）配备独立的管理属性，如标签、任务状态、收藏等。
1.2.2研究意义
本研究不仅旨在开发一个实用的工具，更具有重要的理论价值和实践意义。
在理论贡献方面，本研究深入探索了“卡片式原子化”这一信息管理理念在个人笔记系统中的具体应用和实现路径，为该领域提供了新的轻量化设计思路和实证支持。同时，通过构建一个覆盖 Web、桌面和移动端的复杂应用，本研究在前后端分离架构下的跨平台设计、数据同步策略等方面进行了有益的探索，并对 PWA 技术在信息管理系统中的离线优先架构应用提供了实践参考。在实践意义方面，本系统直面传统笔记应用因采用“文档”模式而导致的信息组织僵化、检索困难等痛点。本研究的“卡片式原子化”设计，避免了传统笔记的信息堆积，使用户能快速捕获和检索单个灵感或任务。这为广大用户，特别是知识工作者、学生和研究人员，提供了一个高效、轻量、安全的个人信息管理工具，显著提升了其在日常灵感记录和任务跟踪方面的效率。
1.3论文研究主要内容
为了实现上述研究目的，本论文将重点围绕以下几个核心技术问题展开研究与实现：
“卡片式”原子化信息模型设计：
这是本研究的核心。重点研究以“笔记卡片”为基本单元的数据模型，设计一个能支持文本、图片、标签（Tagging）、待办（To-do）和收藏（Favorite）等多种状态的独立卡片结构，确保每个“原子”的自洽性。
多平台架构设计与数据同步机制：
设计统一的、可扩展的后端 RESTful API 架构，以支持 Web、桌面（通过 Electron 或 Tauri）和移动端（通过 PWA 或混合应用框架）等多种客户端的接入。研究并实现一个基于事件驱动的实时数据同步机制，包括智能冲突检测和解决方法，确保多设备并发编辑时的数据一致性。
离线优先架构与 PWA 技术应用：
基于 Service Worker、Cache API 和 IndexedDB (通过 Dexie.js 封装) 等 Web 技术，构建 PWA 离线优先架构。实现应用的核心功能离线可用、本地数据的高效存储与查询、以及网络恢复后的后台增量同步。
系统功能实现与交互优化：
基于 Vue 3、TypeScript 和 Vite 等现代化前端技术栈进行组件化开发。集成如 TipTap 这样的所见即所得富文本编辑器，并对其进行定制以支持原子化的块级操作。设计响应式用户界面，适配从手机到桌面不同尺寸的屏幕。
AI 智能功能集成与隐私保护：
探索集成第三方大语言模型（LLM）API，为用户提供智能写作、内容总结、信息分析和标签自动推荐等功能。同时，设计本地优先和细粒度的隐私控制机制，确保用户数据安全，允许用户自主选择哪些数据可用于 AI 处理。
1.4国内外研究现状
个人信息管理（PIM）作为一个持续的研究领域，在国内外都已形成相对成熟的理论体系和产品生态。
1.4.11. 国外研究现状：
国外的研究和产品起步较早，引领了个人信息管理工具的发展。
产品层面： Evernote 作为早期代表，建立了基于“笔记本-笔记”的层级结构，普及了多媒体内容支持和全文搜索。Microsoft OneNote 则采用了更加自由的画布式设计，允许用户在任意位置添加内容。近年来，以 Notion 为代表的新兴力量凭借其模块化（Block）的设计理念和强大的数据库功能，将信息管理的粒度推进到“块”，这与本研究的“原子化”思想有相似之处，但其强依赖云端和复杂的数据库配置也带来了学习成本高和离线体验不佳的问题。Obsidian 和 Roam Research 则专注于通过双向链接构建网状知识库，但在多平台同步和易用性上有所欠缺。
学术层面： 斯坦福大学等研究机构在个人信息管理（PIM）领域进行了深入探索，提出了信息碎片化管理和上下文感知检索等概念。MIT 的研究者则在分布式存储和数据同步技术上有着深厚积累，为多设备协同工作提供了理论基础。
2. 国内研究现状：
国内在个人信息管理系统的研究起步相对较晚，但发展迅速，并在协同办公领域表现出色。
产品层面： 腾讯文档、石墨文档等产品在协同编辑和云端同步方面表现优秀，但其核心定位是“文档”而非个人信息管理。语雀和飞书文档则更多地面向团队和企业用户，构建结构化的团队知识库，其设计哲学更侧重于“发布”而非个人的“收集与思考”。
学术层面： 清华大学、北京大学等高校的研究团队在分布式系统、数据同步算法、用户界面设计等相关领域取得了重要进展。中科院计算所在大数据存储和检索技术方面的研究也为笔记系统的技术实现提供了支撑。
3. 总结与评述：
综合国内外研究现状，当前个人信息管理系统正呈现出向多平台融合、AI 智能处理、分布式和隐私保护演进的趋势。然而，尽管现有研究和产品在功能丰富性方面已相对成熟，但在以下几个关键问题上仍存在普遍的不足：
信息粒度管理缺乏灵活性： 大多数系统仍以传统的文档或页面为基本单位，即便是 Notion 的“块”，其灵活性和可重组性也有限，难以支持真正意义上的“原子化”操作。
跨平台同步的一致性和实时性有待提升： 特别是在网络环境不稳定的情况下，如何保证数据一致性、减少冲突，仍是技术难题。
用户隐私与数据控制权： 多数主流应用采用中心化的云存储方案，用户对自己的数据缺乏完全的控制权，隐私泄露风险较高。

因此，设计一款兼顾信息组织灵活性（原子化）、多平台一致性（PWA 与实时同步）以及用户数据控制权（本地优先）的笔记系统，是当前 PIM 领域一个具有重要研究价值和广阔应用前景的课题。

关键理论与技术介绍
本章将介绍本系统在设计与实现过程中所需要的主要技术以及核心理论基础。这些技术的有机结合，共同构成了系统轻量、高效、跨平台的核心特性。
2.1“卡片式原子化”理论
本研究的核心设计理念是“卡片式原子化”。该理论借鉴了原子化思想，但将其应用粒度从文档内部的“块”转变为“笔记卡片”本身。在传统笔记应用中，信息常以“文档”为单位进行组织，导致单个的灵感、待办或信息片段被淹没在冗长的内容中，难以被独立检索和管理。本系统提出的“卡片式原子化”理论，是将用户的每一次输入，无论是一个想法、一个任务还是一段摘录，都封装成一个独立的、自洽的“信息原子”，即一张笔记卡片。每张卡片都拥有独立的属性，如标签、状态、创建时间等，使其成为信息流中的最小管理单元。这种设计规避了传统笔记的信息堆积问题，极大地提升了轻量级信息（如灵感、待办）的捕获、组织和检索效率。
2.2前后端分离架构
为实现系统的多平台覆盖能力、提升开发效率并降低维护成本，本系统在整体架构上采用了成熟的前后端分离模式。在该架构下，前端（客户端）和后端（服务器）是两个独立部署的项目。前端专注于用户界面、交互逻辑和设备适配；后端则专注于业务逻辑处理、数据持久化和 API 接口提供。两者通过标准化的 RESTful API 进行通信，数据交换格式统一采用 JSON。这种松耦合的架构使得前端可以一次开发，通过 PWA 等技术部署到 Web、桌面和移动端等多个平台，而后端则可以独立迭代和扩展，为多端提供一致的数据服务。
2.3关键技术栈介绍
2.3.1Vue
系统的前端是用户交互的核心，为确保开发的规范性、高效性和代码的健壮性，我们选用了以 Vue 3 为核心的现代化技术栈。Vue 3 是一个渐进式 JavaScript 框架，其引入的 Composition API（组合式 API）相较于
Options API，能更好地组织和复用组件逻辑，尤其适合构建本系统这样具有复杂交互状态的单页应用（SPA）。
为了提升大型项目的可维护性并减少运行时错误，系统全面采用了 TypeScript。TypeScript 是 JavaScript 的一个超集，它增加了静态类型系统。在开发阶段，类型检查能够帮助团队规避大量潜在的类型错误，并为编辑器提供了更智能的代码提示和重构能力，显著提升了代码质量和开发效率。
在构建工具方面，系统选用了 Vite。Vite 是一个基于原生 ES Moudle 的新型前端构建工具，它提供了极速的冷启动时间和近乎瞬时的热模块替换（HMR），极大地优化了开发体验，使开发者能更专注于功能的实现。
2.3.2Node.js
系统的后端服务是支撑所有平台数据同步和业务逻辑的核心。我们选用了 Node.js 作为服务端的运行时环境。Node.js 基于 Chrome V8 引擎，其事件驱动、非阻塞 I/O 的特性，使其非常适合构建高并发、低延迟的数据密集型应用，如本系统的实时同步 API。
在 Node.js 环境下，我们使用 Express.js 框架来快速搭建 RESTful API 服务。Express 是一个简洁而灵活的 Web 应用框架，它提供了丰富的中间件生态系统，使我们能高效地处理路由、请求解析、错误处理等任务。为确保 API 的安全性，系统采用了 JWT（JSON Web Token） 进行用户认证和授权。用户登录成功后，后端会签发一个包含用户信息的 Token，前端在后续请求中携带此 Token，后端通过中间件验证 Token 的有效性，从而保护了用户数据不被未授权访问。
2.3.3MongoDB
在数据持久化方面，系统选用了 MongoDB 作为主数据库。MongoDB 是一款高性能的 NoSQL 数据库，它采用 BSON（一种类 JSON 的二进制格式） 文档模型来存储数据。这种灵活的文档结构与本系统“卡片式原子化”的数据模型（每张卡片是一个独立的对象）高度契合，便于快速迭代和扩展字段。相比传统的关系型数据库，MongoDB 在处理大量非结构化或半结构化的笔记数据时，展现了更优的读写性能和扩展性。
2.3.4PWA
为实现“面向多平台”和“离线可用”的核心目标，本系统采用了 PWA（Progressive Web App，渐进式网络应用） 技术。PWA 允许 Web 应用通过现代浏览器的能力，提供接近原生应用（Native App）的体验，如添加到主屏幕、离线访问和消息推送。
PWA 的核心是 Service Worker。它是一个在浏览器后台独立于页面的脚本，充当了网络请求的代理服务器。本系统利用 Service Worker 拦截网络请求，通过**缓存优先（Cache First）**策略，将已获取的笔记数据和应用外壳（App Shell）存储在本地缓存中。这使得用户在网络环境不佳甚至完全离线时，依然可以访问和创建笔记。
对于离线时产生的新数据或修改，系统将其暂存在浏览器的本地数据库 IndexedDB 中。IndexedDB 是一种非关系型的前端数据库，适合存储大量结构化数据。当网络连接恢复时，Service Worker 会在后台检测到网络状态变化，并将 IndexedDB 中暂存的数据与云端的 MongoDB 数据库进行增量同步，最终确保多端设备的数据一致性。
2.4本章小结
本章首先阐述了作为系统设计灵魂的“卡片式原子化”理论，它定义了以笔记卡片为独立信息单元的管理模式。为实现这一理念，本章介绍了系统所采用的“前后端分离”架构。在此架构基础上，详细分析了前端的 Vue 3 技术栈、后端的 Node.js 服务端技术以及 MongoDB 文档型数据库的选择，论证了它们与“卡片式”数据模型的契合性。最后，本章重点讨论了 PWA 及其核心的 Service Worker 和 IndexedDB 技术，它们是实现系统跨平台覆盖和离线可用性的关键。这些理论与技术的有机结合，为后续章节的系统设计与实现奠定了坚实的基础。

3.1系统概述
本系统全称为“面向多平台的原子化笔记系统”，是一个旨在解决传统笔记应用中信息堆积、检索困难问题的轻量化个人信息管理工具。系统的核心设计哲学是第二章所阐述的“卡片式原子化”理论，即将用户的每一次灵感记录或任务事项都视为一个独立的“笔记原子”（表现为一张卡片）。系统基于前后端分离架构，前端利用 Vue 3 和 PWA 技术构建跨平台用户界面，实现离线可用性；后端则通过 Node.js 和 MongoDB 提供安全、稳定的 API 服务和数据持久化。本系统的目标是为用户提供一个界面极简、启动迅速、多端同步、离线可用的笔记体验，使其能毫无负担地快速捕获和管理个人思绪。
3.2自动驾驶可视化系统的需求分析
需求分析是确保系统符合用户预期和解决实际问题的根本前提。本系统的目标用户群体定位于需要快速记录灵感、管理轻量级任务和待办事项的知识工作者、学生及广大普通用户。基于对这类用户使用场景的分析，本系统的需求分为功能性需求和非功能性需求两大类。
3.2.1功能性需求分析
在功能性需求方面，系统必须围绕“笔记原子”这一核心实体提供完整的生命周期管理。首先，系统应提供用户管理功能，包括用户注册和登录，这是保障数据私密性的基础。其次，核心的笔记管理功能是重中之重，用户必须能够便捷地创建新笔记（原子）、查看笔记列表、编辑已有笔记以及删除笔记。为增强信息组织能力，系统需支持对单个笔记原子进行属性标记，如（根据系统截图所示）标记为“收藏”（收藏☆）、“完成”（待办✓）以及附加“标签”（标签#）的能力。此外，为实现信息的高效触达，系统必须提供一个全局的笔记搜索功能（搜索），允许用户通过关键词快速在所有“原子”中定位所需信息。
3.2.2非功能性需求分析
在非功能性需求方面，系统的设计必须满足以下关键质量属性。第一，跨平台一致性，作为系统标题的核心要求，应用必须能在 Web 网页、桌面应用和移动设备（通过PWA）上提供一致的界面和功能体验。第二，数据实时同步，用户的笔记数据必须在登录后于所有设备间自动、快速地同步。第三，离线可用性，基于 PWA 技术，系统应能在网络中断或弱网环境下正常运行，允许用户离线查看和创建笔记，并在网络恢复后自动同步。第四，高性能与易用性，系统界面必须简洁直观，启动和响应速度快，最大程度降低用户的认知负担和操作成本。第五，安全性，用户数据必须得到妥善保护，前后端通信需加密，用户认证需安全可靠。

3.3.1前端视图层设计概述
前端视图层是用户与系统交互的直接界面，其设计的核心目标是简洁、易用、响应迅速且跨平台一致。本系统前端基于 Vue 3 框架构建为单页应用（SPA），确保了组件化开发的高效性和应用的高性能。
在界面布局上，系统采用了移动优先的“卡片流”设计（如系统截图所示）。顶部是快速笔记创建区域，允许用户快速捕获灵感；中部是核心的笔记卡片列表，每张卡片（即一个“笔记原子”）都清晰地展示其内容和状态属性（如收藏、完成、标签）；底部是固定的选项卡导航栏，包含“首页”、“搜索”和“我的”三个主要入口，提供了清晰的功能分区和便捷的视图切换。
为实现多平台覆盖和离线可用性，前端视图层采用 PWA（渐进式网络应用） 技术进行封装。通过 Service Worker 缓存应用外壳（App Shell）和静态资源，使得应用可以被“安装”到用户的主屏幕，并在离线时快速启动。这种设计不仅统一了 Web、桌面和移动端的访问体验，也为实现离线数据操作提供了基础。
3.3.2数据交互层设计概述
数据交互层定义了前端视图层与后端服务层之间的通信契约和数据流转机制。本系统的前后端完全分离，二者之间通过标准化的 RESTful API 接口进行异步通信。
所有的数据交换均采用轻量级的 JSON 格式。在安全性方面，系统采用 JWT（JSON Web Token） 机制进行用户身份认证。用户登录成功后，服务层会签发一个包含用户身份信息的 Token，前端视图层将其存储在本地（如 LocalStorage），并在后续的每一次 API 请求中，通过 HTTP 的 Authorization 请求头将其发送给服务器，以便服务层进行身份校验。
在离线交互设计上，前端的 Service Worker 会充当网络请求的代理。当用户在离线状态下创建或修改笔记时，数据被暂存在本地的 IndexedDB 数据库中。一旦网络连接恢复，PWA 会在后台自动将这些暂存的数据通过增量同步 API 发送到服务层，从而确保了数据在多端设备间的最终一致性。
3.3.3服务器层设计概述
服务层（后端）是系统的业务逻辑核心，负责处理所有的数据请求、业务规则和用户认证。本系统服务层基于 Node.js 运行时环境和 Express.js 框架搭建，其非阻塞 I/O 特性非常适合处理高并发的数据同步请求。
根据功能需求，服务层被划分为几个核心的功能模块：
用户模块（User Module）：负责处理用户的注册、登录请求。登录成功后，此模块将使用 JWT 签发 Token，并负责后续所有接口的 Token 校验。
笔记管理模块（Note Module）：这是系统的核心业务模块，提供了针对“笔记原子”（Note Card）的完整 CRUD（创建、读取、更新、删除）操作接口。例如，创建新笔记、获取笔记列表、更新笔记内容、切换“收藏”或“完成”状态等。
搜索模块（Search Module）：提供全局笔记搜索的 API 接口。它接收前端传递的关键词，在数据库中执行查询，并返回匹配的笔记卡片列表。
这些模块共同构成了稳定、安全的 API 服务，为所有前端平台提供统一、一致的业务逻辑处理。
3.3.4数据库层设计概述
数据库设计是实现“卡片式原子化”理论的数据基石。本系统选用 MongoDB 作为持久化数据库，其面向文档的存储模型与本系统的业务数据结构高度契合。数据库主要包含以下两个核心集合（Collections）：

users 集合：用于存储用户信息。每个文档代表一个注册用户，核心字段包括 username（用户名，唯一）和 password（存储加密后的密码摘要），用于用户登录验证。

notes 集合：这是系统的核心数据表，该集合中的每一个文档（Document）即对应一个“笔记原子”（一张卡片）。这种设计完美实现了“笔记即原子”的理念。根据需求分析，notes 集合的文档结构（Schema）设计如下：

_id: 文档的唯一标识，由 MongoDB 自动生成。

userId: 用户 ID（外键，关联 users 集合），用于标识该笔记的归属，实现数据隔离。

content: 笔记的文本内容。

isFavorite: 布尔值（Boolean），标记是否为“收藏”状态。

isDone: 布尔值（Boolean），标记是否为“完成”状态。

tags: 字符串数组（Array[String]），用于存储用户附加的标签，便于分类和搜索。

createdAt: 创建时间戳，用于笔记排序。

updatedAt: 最后更新时间戳。

这种以 notes 文档为原子的设计，使得对单条笔记的增、删、改、查操作都极为高效，且易于未来扩展新的属性。
